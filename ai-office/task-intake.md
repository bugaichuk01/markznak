# Единый шаблон постановки задачи

Все задачи формулируются в иерархии **EPIC → Story → Task**. Заполнять до начала разработки. Утверждает тимлид.

**Кому отдавать постановку:** только **Team Lead** (`ai-office/agents/team-lead/role.mdc`). Пользователь не ставит задачи программисту и тестировщику напрямую.

---

## 1. EPIC (крупная область / система)

| Поле | Описание |
|------|----------|
| **Название** | |
| **Цель системы** | Что создаётся на уровне продукта |
| **Бизнес-задача** | Какую проблему решаем |
| **Архитектурная роль** | API / worker / adapter / persistence / integration / client / reporting — что уместно для продукта |
| **Основные компоненты** | Список |
| **Стек** | Python, FastAPI, Redis, PostgreSQL, Docker, … (по факту задачи) |
| **Границы и зависимости** | Внешние API, очереди, клиенты, ограничения по данным (PHI и т.д.) — если применимо |
| **Взаимодействие с другими сервисами** | Кто вызывает, кто вызывает нас, формат данных |

---

## 2. Story (функция внутри EPIC)

Формат: **As a** \<actor\> **I want** \<capability\> **So that** \<result\>.

| Поле | Описание |
|------|----------|
| **Название** | |
| **As a** | Роль (оператор, врач, система, интеграция МИС, …) |
| **I want** | Возможность |
| **So that** | Результат/ценность |
| **Описание** | Детали функции |
| **Входные данные** | |
| **Выходные данные** | |
| **Ограничения** | |
| **Зависимости** | Другие Story, сервисы, API |
| **Acceptance criteria** | Чеклист «сделано», когда всё выполнено |

---

## 3. Tasks (инженерные шаги)

Каждый Task — одна конкретная, измеримая работа.

| # | Task (формулировка) | Ожидаемый результат | Место в коде/сервисе | Критерий завершения |
|---|---------------------|---------------------|----------------------|---------------------|
| 1 | | | | |
| 2 | | | | |
| 3 | | | | |

**Требования к Task:**

- Техническая формулировка (не «сделать сервис», а «добавить endpoint POST /api/v1/…», «расширить модель Job», «добавить шаг пайплайна в worker»).
- Понятно, где в репозитории это делается.
- Чёткий критерий завершения (тесты проходят, контракт API соблюдён, миграция/схема согласована с политикой проекта и т.п.).

---

## 4. Дополнительно (при необходимости)

| Раздел | Содержание |
|--------|------------|
| **Архитектура** | Схема компонентов, потоки данных |
| **Структура проекта** | Новые папки, модули, сервисы |
| **API-контракты** | Метод, путь, тело запроса/ответа, коды ошибок |
| **Формат данных** | JSON-схемы, типы, примеры |
| **Риски и ограничения** | Производительность, совместимость, миграции, приватность данных |

## Пример заполнения (коротко)

**EPIC:** Medical Voice Ingestion — приём аудио диктовки, пайплайн обработки, выдача статуса job.

**Story:** Intake API  
As a *клиент записи* I want *загрузить аудиофайл* so that *запускается обработка и я получаю job_id*.

**Tasks:**

1. Реализовать или расширить `POST /api/v1/intake` по контракту.
2. Сохранить файл и создать запись Job в БД.
3. Поставить задачу в очередь (Arq) при наличии Redis.
4. Обновить `.env.example` и README при новых переменных.

После заполнения и утверждения использовать при разработке и отмечать выполненные Task/Story.
